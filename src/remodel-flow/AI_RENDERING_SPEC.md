## 1. Overview

### Purpose
AI rendering helps users visualize their selected bathroom remodel before
committing to it. The goal is to convert plan data (style, package, products)
into an emotionally resonant preview that increases user confidence.

### Preview tiers
There are three conceptual levels of preview fidelity:

- **Concept preview** â€” generic, style-driven illustration of the chosen
  package. No user photo required. Fastest, cheapest, free.
- **Personalized preview** â€” user uploads a photo of their bathroom; the AI
  composes the selected products and style on top of that real space. Paid.
- **Realistic preview** â€” based on a 3D scan of the user's bathroom for
  accurate geometry, lighting, and product placement. Premium tier.

All three share the same `RenderRequest` contract.

## 2. RenderRequest structure

Defined in `src/remodel-flow/render.ts`. Fields currently used in code:

| Field | Type | Notes |
|---|---|---|
| `render_session_id` | `string` | Unique per request (UUID). |
| `mode` | `"template" \| "photo" \| "scan"` | See **Render modes** below. |
| `render_intent` | `"concept"` | Preview purpose. Reserved for future intents. |
| `variation_index` | `number` | Which variation of this request (0 = first). |
| `selected_package_id` | `string \| undefined` | The chosen package id. |
| `selected_style` | `StyleId \| undefined` | The chosen style. |
| `selected_tier` | `TierId \| undefined` | The chosen tier. |
| `resolved_state.slots` | `ResolvedSlot[]` | Engine-resolved slots â€” **not** flattened product bins. |
| `style_profile` | `{ style?, descriptors[] }` | Style + free-form descriptors derived from selections. |
| `bathroom_size_template` | `"small" \| "medium" \| "large" \| "unknown"` | Coarse size hint. |

`resolved_state` is intentionally passed as `{ slots }` only. Slot shape
contains the canonical product reference (option id, option name, category)
the renderer should use for product layering.

## 3. Render modes

| Mode | Tier | Input | Status |
|---|---|---|---|
| `template` | Free | Selections only | Active (UI only, no AI call yet) |
| `photo` | Paid (future) | Selections + user photo | Planned |
| `scan` | Premium (future) | Selections + 3D scan | Planned |

Mode determines which prompt layers and reference inputs are composed.

---

## 4. Prompt composition (high level)

Prompts are built from four stacked layers. Lower layers establish the scene;
higher layers refine it.

1. **Base room layer** â€” derived from `bathroom_size_template` and `mode`.
   In `photo`/`scan`, the user's actual room replaces the generic base.
2. **Style layer** â€” derived from `selected_style` and `style_profile.descriptors`
   (materials, finishes, tones).
3. **Product layer** â€” derived from `resolved_state.slots`: vanity, tile,
   fixtures, lighting, etc., referenced by option name and category.
4. **Constraint layer** â€” guardrails: keep architectural lines, respect room
   geometry, no extra rooms/fixtures, plausible scale, neutral framing.

`render_intent` and `variation_index` modulate phrasing and seed selection.

## 6. Free vs paid logic (high level)

| Plan | Allowed modes | Quality |
|---|---|---|
| **Free** | `template` only | Standard |
| **Paid** | `template` + `photo` | Higher quality, more variations |
| **Premium** | `template` + `photo` + `scan` | Highest quality, geometry-accurate |

Gating happens at request time based on the user's plan. The `RenderRequest`
contract itself is plan-agnostic.
